# Git 规范

## 工作区安全

- 开始修改前检查 `git status`，识别已有未提交改动。
- 不撤销、覆盖或格式化与当前任务无关的用户改动。
- 禁止使用 `git reset --hard`、强制推送或其他破坏性操作，除非用户明确要求。
- 不提交 `.env`、凭据、构建产物、缓存或与任务无关的文件。

## 提交规范

- 每个提交应聚焦一个可解释的变更。
- 提交前检查 diff，并运行与改动匹配的验证命令。
- 提交消息使用 Conventional Commits 风格：

```text
<type>(optional-scope): <summary>
```

- 常用类型：`feat`、`fix`、`docs`、`test`、`refactor`、`chore`。
- 摘要使用祈使语气，简洁说明实际变化，不使用模糊描述。
- 未经用户明确要求，不主动创建提交、推送分支或重写历史。

## 分支与合并

- 保持分支改动范围聚焦，避免混入无关修改。
- 解决冲突时保留并理解双方有效改动，不盲目选择任一侧。
- 合并或提交前确认工作区状态与预期一致。
